# Phase 2.1: Domain Consolidation - Context

**Gathered:** 2026-02-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Merge scattered server modules into cohesive domain folders. No new functionality — purely organizational. The goal is that every source file lives in the domain it belongs to, following the same internal structure established in Phase 2. After this phase, the server/src/ tree is clean and domain-oriented before auth is layered on in Phase 3.

</domain>

<decisions>
## Implementation Decisions

### Domain groupings
- `discovery/` absorbs `graph/`, `dcat/`, `rdf/` — all graph/discovery/catalog functionality under one domain, renamed from "graph" to "discovery" for broader scope
- `validation/` — Claude inspects the code to determine: if validation validates graphs/catalogs → merge into `discovery/`; if it validates connections → merge into `connections/`
- `jobs/` absorbs `pipeline/`, `script/` — pipeline executes inside a job, script rewrites are job operations
- `ai/` absorbs `conversations/` — conversations are the AI chat feature
- `cloud_accounts/` absorbs `cloud/` — cloud provider abstraction (reader, resolver) belongs with cloud account management

### Folder naming
- The graph+dcat+rdf group is named `discovery/` (not `graph/`)
- All other domain folders keep their current names

### Loose routes cleanup
- Claude's discretion on where `health.rs`, `providers.rs`, `scripts.rs` go, based on what they actually do
- Goal: `routes/` should either be empty (deleted) or contain only truly cross-cutting route infrastructure (like `build_router()`)

### Internal file structure
- All merged domains follow the same pattern as Phase 2: `mod.rs`, `routes.rs`, `db.rs`, `models.rs`, `errors.rs`
- Only include files that are actually needed (no empty placeholder files)
- If a domain has no DB access, skip `db.rs`; if no errors, skip `errors.rs`

### Claude's Discretion
- Where validation/ ends up (graph or connections) based on code inspection
- Exact placement of health.rs, providers.rs, scripts.rs
- Internal submodule organization within merged domains
- Whether routes/mod.rs (build_router) stays or moves

</decisions>
