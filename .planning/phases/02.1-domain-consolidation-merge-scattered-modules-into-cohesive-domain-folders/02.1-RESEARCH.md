## Summary

This phase is a pure Rust module reorganization with no behavior changes. The server currently has 20 top-level modules in `server/src/`. Several are scattered fragments that belong inside already-established domain folders. After code inspection, the consolidation map is clear and complete.

The main technical challenge is correctly updating every `use` path and `mod` declaration after files move. Rust's module system is path-based — every file that imports from a moved module needs its `use crate::old_path` updated to `use crate::new_path`. The second challenge is updating `main.rs` to remove the now-internal module declarations, since only top-level modules are declared there.

**Primary recommendation:** Move files in domain batches (one domain per plan task), verify `cargo build` compiles cleanly after each batch, then run `cargo clippy` at the end. Order moves from least-referenced to most-referenced to minimize the number of import fixups needed per step.

### Validation Module Placement Decision (Claude's Discretion)

**Verdict from code inspection: merge into `discovery/`**

Evidence:
- `validation/mod.rs` contains `ValidatableGraph` — it validates RDF graphs using ShEx/SHACL shapes
- The input to validation is a `data_url` (RDF file) and a `shape_path` (RDF shapes file)
- It uses `rudof_rdf`, `shacl_validation`, `shex_validation` — RDF/semantic web libraries
- It references `cloud/reader` to download the RDF data, and `connections/models::LocationType` for the connection type
- The route is `/v1/validate` — validates an RDF dataset against a shape schema
- Nothing about this touches connection CRUD or connection infrastructure; it *uses* a connection reference to find the cloud URL but its concern is graph validation

This is graph/discovery-domain functionality (validating that an RDF graph conforms to a shape schema), not connection-domain. It belongs in `discovery/`.

Final placement:
- `validation/mod.rs` → `discovery/validation.rs` (the `ValidatableGraph` struct and its methods)
- `validation/routes.rs` → merged into `discovery/routes.rs` or kept as `discovery/validation_routes.rs`
- `validation/types.rs` → `discovery/validation_types.rs`
- Route registered at `/v1/validate` stays exactly as is in `routes/mod.rs`

### Loose Routes Placement Decision (Claude's Discretion)

**`routes/health.rs`** — Keep in `routes/`. It is purely cross-cutting infrastructure (liveness/readiness checks). Has no domain home.

**`routes/providers.rs`** — Keep in `routes/`. It calls `jobs::script::init_context()` to introspect fossil-lang providers. It is a cross-cutting utility endpoint, not a jobs entity. No domain home. Route: `GET /v1/providers`.

**`routes/scripts.rs`** — Keep in `routes/`. It calls `jobs::pipeline::*` and `jobs::script::*` — it is the script validation handler at `POST /v1/scripts/validate`. This is a cross-cutting utility endpoint.

**Conclusion:** `routes/` keeps `mod.rs` (build_router), `health.rs`, `providers.rs`, and `scripts.rs`. It is no longer a home for domain routes — those all live in their domain — but it retains genuinely cross-cutting infrastructure.

### Pattern: Dissolving a Module Into a Destination Domain

When moving `cloud/` into `cloud_accounts/`:

**Step 1 — Copy files:**
```
cloud/reader.rs  →  cloud_accounts/reader.rs
cloud/resolver.rs  →  cloud_accounts/resolver.rs
```

**Step 2 — Update internal `use super::` references in moved files:**
```rust
// Before (cloud/reader.rs):
use super::build_store;
use super::is_cloud_url;

// After (cloud_accounts/reader.rs):
use super::build_store;    // still correct — super is now cloud_accounts
use super::is_cloud_url;   // still correct
```

**Step 3 — Move `cloud/mod.rs` logic into `cloud_accounts/mod.rs`:**
The `cloud/mod.rs` contains `build_store()`, `is_cloud_url()`, `is_data_path()`. These become `pub` functions exposed from `cloud_accounts`.

**Step 4 — Update all callers:**
Search for `crate::cloud::` references and rewrite to `crate::cloud_accounts::`.

**Step 5 — Remove `mod cloud;` from `main.rs`.**

### Pattern: Absorbing a Sub-module Into an Existing Domain

When absorbing `pipeline/` into `jobs/`:

The `pipeline/mod.rs` exports:
- `pub use extract::extract_summary;`
- `pub use types::*;` (Field, PipelineSummary, ValidationResult, etc.)
- `ProgramQuery` struct

These need to become accessible as `crate::jobs::pipeline_*` or as direct `crate::jobs::` re-exports.

**Preferred approach:** Use flat files within `jobs/`:
- `jobs/pipeline_types.rs` — the types from `pipeline/types.rs`
- `jobs/pipeline_extract.rs` — the extract functions from `pipeline/extract.rs`
- `jobs/script.rs` — the compile/execute/init_context from `script/mod.rs`
- `jobs/rewrite.rs` — the rewrite module from `script/rewrite.rs`

Then in `jobs/mod.rs`, add:
```rust
pub mod pipeline_extract;
pub mod pipeline_types;
pub mod script;
pub mod rewrite;

// Re-export for callers that used crate::pipeline::*
pub use pipeline_types::{PipelineSummary, ValidationResult, Field, PipelineOutput, /* ... */};
pub use pipeline_extract::extract_summary;
```

Callers update from `crate::pipeline::` to `crate::jobs::`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Finding all callers of a moved path | Manual grep | `grep -rn "crate::graph::"` systematically | Rust compiler will catch misses anyway |
| Verifying no behavior changes | Custom test harness | `cargo build && cargo clippy -- -D warnings` | The compiler IS the test for a pure rename refactor |
| Managing circular dependencies after merge | Architectural redesign | Keep cross-cutting modules (cloud_accounts infra, rdf_format) at mod level and import from there | Rust enforces no cycles; structure already avoids them |

**Key insight:** Rust's type checker is the primary correctness tool for this refactor. Any missed `use` path update is a compile error. The refactor is correct when `cargo build` passes with zero errors.

## Common Pitfalls

### Pitfall 1: `use super::` in Moved Files

**What goes wrong:** Files in `cloud/` use `use super::build_store;` to call functions in `cloud/mod.rs`. After moving to `cloud_accounts/`, `super` is now `cloud_accounts`. This works IF the function is also in `cloud_accounts/mod.rs`. But if the function was left behind or not re-exported, it's a compile error.

**Why it happens:** Moving a file changes what `super::` resolves to.

**How to avoid:** For every moved file, check all `use super::*` references and verify the destination `mod.rs` exposes everything they reference.

**Warning signs:** Compile error `use of undeclared module 'super::X'` immediately after moving a file.

### Pitfall 2: Forgetting `main.rs` Module Declarations

**What goes wrong:** `main.rs` declares `mod cloud;`, `mod graph;`, `mod pipeline;`, `mod rdf;`, `mod script;`, `mod conversations;`. After dissolving these modules, their declarations in `main.rs` must be removed. Leaving them produces "file not found for module" errors.

**Why it happens:** Rust's module system requires an explicit `mod foo;` declaration to include a file. Deleting the file without removing the declaration is an error.

**How to avoid:** At the end of each domain merge, remove the dissolved module's `mod` declaration from `main.rs`.

**Warning signs:** `error[E0583]: file not found for module 'cloud'`.

### Pitfall 3: Pub Visibility Gaps When Absorbing

**What goes wrong:** `pipeline/types.rs` exports `pub struct PipelineSummary`. When moved to `jobs/pipeline_types.rs`, the struct is `pub` within the file but not re-exported from `jobs/mod.rs`. Callers using `crate::pipeline::PipelineSummary` find the new path `crate::jobs::pipeline_types::PipelineSummary` but the old callers still use the old path.

**Why it happens:** Moving a file into a submodule adds a path segment. Old callers need updating.

**How to avoid:** After each move, run `cargo build` to surface all broken imports. Fix systematically.

**Warning signs:** `error[E0432]: unresolved import 'crate::pipeline::PipelineSummary'`.

### Pitfall 4: Circular Import After Merge

**What goes wrong:** `conversations/routes.rs` uses `crate::ai::client::ask_llm`. After `conversations` is absorbed into `ai`, and `routes.rs` is now `ai/routes.rs`, the `use crate::ai::client::ask_llm` becomes a self-referential import: `use crate::ai::client::ask_llm` from within `ai/routes.rs` is fine — it's just a sibling module. No circularity here.

**Why it happens:** This pitfall sounds scary but in practice doesn't apply here since `conversations/routes.rs` already imports from `ai::client` — the merge just makes this a sibling import.

**How to avoid:** After merge, replace `use crate::ai::client` with `use super::client` (or keep the full path — both work).

### Pitfall 5: `OutputCache` in `main.rs` References `graph::rdf_graph::RdfGraph`

**What goes wrong:** `main.rs` contains:
```rust
use graph::rdf_graph::RdfGraph;
```
and the `OutputCache` struct holds `Arc<RdfGraph>`. After renaming `graph/` to `discovery/`, this must be updated to:
```rust
use discovery::rdf_graph::RdfGraph;
```

**Why it happens:** `main.rs` is a direct consumer of the renamed module.

**How to avoid:** Include `main.rs` in the systematic `use` path update pass.

### Pitfall 6: `validation/routes.rs` uses `crate::error::error_body` (old pattern)

**What goes wrong:** `validation/routes.rs` still uses the old `crate::error::error_body` helper and manual `StatusCode` tuples — it was not updated to the Phase 2 domain-error pattern. This is intentional (no behavior changes), but it means the moved file will continue using the old error style. This is acceptable — the user said no behavior changes, and error response style is behavior.

**How to avoid:** Do not attempt to modernize error handling in this phase. Leave the file as-is aside from path updates.

## Recommended Task Order

Tasks should be ordered from least-referenced to most-referenced to reduce cascading fixup work:

1. **`ai/` absorbs `conversations/`** — `conversations` has 6 callers in `routes/mod.rs` + `main.rs`. Moving first is clean because `ai` already exists and `conversations/routes.rs` already imports from `crate::ai::client`.

2. **`cloud_accounts/` absorbs `cloud/`** — `cloud` is referenced from `jobs/runner.rs`, `graph/routes.rs`, `validation/routes.rs`, and `script/rewrite.rs`. Moving cloud before graph simplifies graph's internal fixups.

3. **`graph/` becomes `discovery/` + absorbs `dcat/`, `rdf/`, `validation/`** — This is the largest merge. `graph` becomes the foundation domain folder renamed to `discovery`. `dcat`, `rdf`, and `validation` are absorbed into it. All their internal cross-references become sibling imports.

4. **`jobs/` absorbs `pipeline/` and `script/`** — `pipeline` is referenced by `conversations/routes.rs`, `routes/scripts.rs`. Since conversations is already moved to `ai`, there is one fewer external caller by this point. `script` is referenced by `jobs/runner.rs` (already in `jobs`), `routes/providers.rs`, and `routes/scripts.rs`.

5. **Final cleanup** — Remove dissolved `mod` declarations from `main.rs`. Verify `cargo build && cargo clippy`.

   - What we know: `discovery/` will contain files from `graph/` (4 files), `dcat/` (3 files), `rdf/` (1 file), and `validation/` (3 files) = 11 source files plus `mod.rs`
   - What's unclear: Whether to use a flat file layout (all in `discovery/*.rs`) or nested submodules (`discovery/dcat/`, `discovery/validation/`)
   - Recommendation: Flat layout. The files are small and the current codebase avoids deep nesting. Flat files with prefixes (`dcat_extract.rs`, `dcat_generator.rs`) keep the module simple and match the style of `jobs/runner.rs` alongside `jobs/db.rs`.

2. **Re-export strategy for `jobs/` public surface**
   - What we know: `routes/scripts.rs` currently imports `use crate::pipeline::{PipelineSummary, ValidationResult, extract_summary};`
   - What's unclear: Should `jobs/mod.rs` re-export these types at the `jobs::` level, or should callers use `jobs::pipeline_types::PipelineSummary`?
   - Recommendation: Re-export from `jobs/mod.rs` (`pub use pipeline_types::PipelineSummary;` etc.) so external callers use `crate::jobs::PipelineSummary`. This mirrors how `pipeline/mod.rs` currently does `pub use types::*`.

3. **`cloud_accounts/mod.rs` organization after absorbing `cloud/`**
   - What we know: `cloud/mod.rs` contains `build_store()`, `is_cloud_url()`, `is_data_path()` plus `pub mod reader; pub mod resolver;`
   - What's unclear: Whether these utility functions go inline in `cloud_accounts/mod.rs` or into a dedicated `cloud_accounts/cloud.rs`
   - Recommendation: Inline into `cloud_accounts/mod.rs`. They are small (3 functions + macro) and `mod.rs` is the natural home for domain-level utilities.
