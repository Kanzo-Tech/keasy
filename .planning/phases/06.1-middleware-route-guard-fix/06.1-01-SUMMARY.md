# Phase 06.1 Plan 01: Middleware Route Guard Fix Summary

**One-liner:** Two-list Next.js route guard with ALWAYS_PUBLIC_PATHS for /invite and redirect-after-login via ?redirect= param with open-redirect prevention.

## What Was Built

Fixed two integration gaps from Phase 6:

1. **proxy.ts rewrite** — Replaced the single `PUBLIC_PATHS` list with a two-list pattern:
   - `AUTH_REDIRECT_PATHS = ["/login", "/register"]` — unauthenticated users may access; authenticated users redirect to `/`
   - `ALWAYS_PUBLIC_PATHS = ["/invite"]` — accessible regardless of auth state (critical for invite registration flow)
   - Extracted `matchesPath()` helper for clean, reusable path matching
   - Unauthenticated access to protected routes now sets `?redirect=<pathname>` using `URLSearchParams.set()` API

2. **login/page.tsx** — After successful login, reads `?redirect=` param and navigates to validated destination instead of always going to `/`:
   - Validates redirect starts with `/` to prevent open redirect attacks
   - Falls back to `/` if no redirect param or if redirect is an absolute URL
   - No UI changes, no Suspense boundary changes

## Decisions Made

- **Do NOT rename proxy.ts**: The plan's CONTEXT.md mentions renaming to middleware.ts, but RESEARCH.md documents that this Next.js version (16.1.6) requires `proxy.ts` with `export function proxy()`. The file was NOT renamed.
- **Always-public check runs first**: Order matters — ALWAYS_PUBLIC_PATHS check precedes AUTH_REDIRECT_PATHS check to ensure `/invite` is never intercepted.
- **Validation on consuming side**: Open redirect prevention is handled in login/page.tsx, not in proxy.ts — consistent with CONTEXT.md locked decisions.
